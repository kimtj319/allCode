# 68. 한글 입력 / 입력창 디자인 조사 (2026-06-14)

## ① 한글 입력 조사
실 PTY로 한글(IME 커밋 형태 = 완성형 UTF-8 음절)을 입력해 검증.

- **조합 완료 입력은 정상**: "안녕하세요 한글 테스트입니다" 표시·제출 프롬프트 일치, 백스페이스
  음절 단위 동작, 좁은 폭(40칸)에서 더블폭 wrapping도 38+2=40 경계로 정확(오버플로 0).
- **발견한 실제 버그 — 입력 caret off-by-one**: 입력 후 커서 컬럼이 `텍스트 끝 + 1`에
  위치했다. 프롬프트 prefix `› `(=2칸, `›`는 east-asian width N이라 1칸)인데
  `_render_state`의 cursor base가 3이어서 화면 컬럼 = `1+3+W = 4+W`로, 텍스트 끝(`3+W`)보다
  한 칸 오른쪽. ASCII에도 동일하지만 **한글(더블폭) + IME 조합 중에는 글자가 한 칸 밀려
  보이는 차이**로 더 잘 드러난다. → 가장 유력한 "한글이 달라 보인다"의 원인.
  - 수정: `_PROMPT_PREFIX_WIDTH=2`로 base 교정. 실측 재확인:
    hello world(W11)→14, 안녕하세요(W10)→13, "안녕…입니다"(W28)→31 = 모두 `3+W` 정확.
- **재현 한계**: 실제 macOS IME의 *조합 미리보기*(자모가 음절로 합쳐지는 과정)는 OS/터미널
  레벨이라 PTY 바이트 주입으로 재현 불가. caret 정합 수정으로 조합 글자 위치가 올바르게
  되어 체감이 개선될 것으로 보이나, 다른 증상(예: 자모가 따로 보임)이라면 실 터미널에서
  관찰된 구체 증상이 추가로 필요.

## ② 입력창 디자인 (vs Codex v0.139.0) — 실 PTY 캡처 비교
- **구조는 동일**: 양쪽 모두 상단 헤더 박스(╭─╮: 로고/model/directory) + `› <입력>` 마커 +
  하단 상태줄. **입력을 테두리 박스로 감싸지 않는다**(둘 다). 입력 마커 `›` 동일.
- **미세 차이(코스메틱)**:
  - Codex는 헤더와 입력 사이에 `Tip: ...` 안내 한 줄을 둠. allCode는 없음.
  - Codex model 줄에 `/model to change` 힌트 inline. allCode는 없음.
  - 상태줄 표기: Codex `gpt-5.5 xhigh · <dir>` vs allCode `model: … · workspace: … · approval: …`.
- 결론: 입력창의 핵심 디자인(마커·박스 유무)은 Codex와 **동일**. 차이는 안내문/상태줄 문구
  수준이며 기능 격차 아님. (caret off-by-one만 실제 버그였고 수정함.)
