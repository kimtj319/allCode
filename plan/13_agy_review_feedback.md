# 13. agy 검토 피드백 반영 요약

## 검토 목적

초기 계획 문서 묶음을 GPT-5.5 같은 모델에게 한 번에 구현 요청으로 전달했을 때 정상적으로 완주 가능한지 agy에 검토시켰고, 그 피드백을 계획 문서에 반영했다.

## agy 검토 결론

현재 계획서는 엔터프라이즈급 아키텍처 방향은 적절하지만, 한 번에 전체 구현을 요청하면 다음 위험이 크다.

1. 출력 토큰 한계로 인해 `pass`, `TODO`, 생략 구현이 발생할 수 있다.
2. Textual TUI와 agent loop의 async lifecycle 경계가 모호하면 UI deadlock 또는 이벤트 누락이 발생할 수 있다.
3. LLM 스트림 파서가 불완전한 tool call delta와 malformed JSON을 견디지 못하면 loop가 붕괴할 수 있다.
4. `patch_file` 포맷이 명확하지 않으면 잘못된 위치 수정 또는 패치 실패가 반복될 수 있다.
5. 대형 코드베이스 컨텍스트를 전체 본문으로 누적하면 모델 품질이 급격히 하락할 수 있다.
6. 자가 수리 루프에 rollback과 반복 제한이 없으면 무한 수정 또는 상태 훼손이 발생할 수 있다.

## 반영한 보강 내용

- 초기 구현 대상 문서에 `구현 전 필수 보강 지시` 섹션을 추가했다.
- `12_mvp_execution_plan.md`를 추가하여 한 번에 구현하더라도 내부적으로 마일스톤 순서를 강제하도록 했다.
- MVP 범위, 제외 범위, 마일스톤별 생성 파일, 검증 명령, 실패 차단 조건을 추가했다.
- 대규모 프로젝트 생성 시 skeleton-first, completion checker, validation, repair, rollback, final report 절차를 강화했다.

## 최종 판단

보강 전에는 전체 구현 요청이 스켈레톤 수준에서 멈출 가능성이 높았다. 보강 후에는 `12_mvp_execution_plan.md`를 함께 전달한다는 조건에서, GPT-5.5가 한 번에 구현을 시도하더라도 내부적으로 단계별 구현과 검증을 수행할 가능성이 높아졌다. 다만 실제 실행 시에는 마일스톤 단위로 검증 로그를 확인하는 방식이 가장 안전하다.
