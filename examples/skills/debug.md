# 체계적 디버깅

증상에서 출발해 근본 원인까지 가설-검증으로 좁힌다. 코드를 무작정 고치지 않고,
원인을 재현·격리한 뒤에 수정한다.

## 절차
1. **재현**: 버그를 안정적으로 재현하는 최소 케이스를 만든다. 재현이 안 되면
   먼저 재현 조건(환경/입력/타이밍)을 찾는다.
2. **관찰**: 정확한 증상을 수집한다 — 에러 메시지, 스택 트레이스, 실제 vs 기대
   동작, 최근 변경(`git log`, `git bisect` 고려).
3. **가설**: 원인 후보를 1~3개 세운다. 각 가설은 검증 가능해야 한다.
4. **격리**: 로그/프린트/디버거/이분 탐색으로 가설을 좁힌다. 한 번에 한 변수만
   바꾼다. 입력→출력 경계를 따라 잘못된 값이 처음 나타나는 지점을 찾는다.
5. **확인**: 근본 원인을 한 문장으로 설명할 수 있어야 한다. "왜 이 입력에서
   이 코드가 이 값을 내는가"가 명확해질 때까지 좁힌다.
6. **수정**: 최소 변경으로 고치고, 그 버그를 재현하던 케이스를 테스트로 박제한다.
7. **회귀 확인**: 관련 테스트 전체를 돌려 부작용이 없는지 본다.

## 원칙
- 증상이 아니라 원인을 고친다. 같은 부류의 다른 입력도 막히는지 확인한다.
- "아마", "분명히" 같은 추측을 증거로 대체한다. 모르면 로그를 더 본다.
- 우연히 고쳐진 것 같으면 왜 고쳐졌는지 설명되기 전까지 끝난 게 아니다.
