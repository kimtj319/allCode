# PR 설명 작성

리뷰어가 빠르게 맥락을 잡도록 PR 제목과 본문을 작성한다. 먼저 base 대비 변경을
읽는다: `git diff main...HEAD --stat`와 커밋 로그(`git log main..HEAD`).

## 제목
- Conventional Commits 스타일 한 줄. 예: `feat(search): 퍼지 파일 검색 추가`.
- 무엇이 바뀌는지 한눈에. 70자 이내.

## 본문 템플릿
```markdown
## 무엇을 (What)
이 PR이 추가/변경/제거하는 것을 2~4줄로.

## 왜 (Why)
해결하는 문제나 동기. 관련 이슈 링크(Closes #N).

## 어떻게 (How)
주요 설계 결정과 접근. 리뷰어가 알아야 할 트레이드오프.

## 테스트 (Testing)
- 어떤 테스트를 추가/수정했는지
- 어떻게 검증했는지 (명령어/결과)

## 체크리스트
- [ ] 테스트 통과
- [ ] 문서 갱신 (필요 시)
- [ ] 호환성 깨짐 없음 (있으면 명시)
```

## 규칙
- 변경 사실만 적는다. 하지 않은 일을 했다고 쓰지 않는다.
- 스크린샷/로그가 도움이 되면 포함을 제안한다.
- 큰 PR이면 리뷰 순서(어느 파일부터 볼지)를 안내한다.
