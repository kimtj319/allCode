# 코드 리뷰

변경분(diff 또는 지정한 파일)을 아래 순서로 점검하고, 발견 사항을 심각도와
함께 보고한다. 칭찬보다 실행 가능한 지적에 집중한다.

## 1. 먼저 맥락 파악
- 변경의 의도가 무엇인지 (PR 설명/커밋 메시지/이슈) 확인한다.
- `git diff`로 실제 변경 범위를 읽는다. 변경되지 않은 코드는 리뷰 대상이 아니다.

## 2. 점검 항목
- **정확성**: 경계 조건(빈 입력, null, 0, 음수, 오버플로), off-by-one,
  비동기 경쟁 조건, 예외 경로에서의 자원 누수.
- **보안**: 신뢰할 수 없는 입력의 검증/이스케이프, 인젝션(SQL/명령/경로),
  하드코딩된 시크릿, 권한 검사 누락, 안전하지 않은 역직렬화.
- **에러 처리**: 삼켜진 예외, 무의미한 로그, 사용자에게 노출되는 내부 정보.
- **성능**: 불필요한 반복/할당, N+1 쿼리, 큰 데이터의 동기 처리, 캐시 무효화.
- **테스트**: 새 동작에 대응하는 테스트가 있는가, 실패 경로도 덮는가.
- **가독성/유지보수**: 명확한 네이밍, 죽은 코드, 중복, 주변 코드 컨벤션 일치.

## 3. 보고 형식
각 항목을 다음처럼 정리한다.
- `[심각도] 파일:줄 — 문제 요약`
  - 왜 문제인지 한 줄.
  - 제안하는 수정 방향(가능하면 구체적 코드).
심각도는 `blocker` / `major` / `minor` / `nit` 중 하나를 쓴다. 문제가 없으면
"치명적 이슈 없음"이라고 명확히 밝히고 사소한 개선만 nit으로 남긴다.
