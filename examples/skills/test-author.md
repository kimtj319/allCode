# 테스트 작성

구현 디테일이 아니라 관찰 가능한 행위를 검증하는 테스트를 쓴다. 먼저 저장소의
테스트 컨벤션(프레임워크, 디렉터리, 네이밍, 픽스처)을 읽고 그에 맞춘다.

## 무엇을 테스트할까
- **해피 패스**: 대표적인 정상 입력 1~2개.
- **경계**: 빈 값, 단일 원소, 최대/최소, 0/음수, 경계 인덱스.
- **실패 경로**: 잘못된 입력에서 올바른 예외/에러를 내는지.
- **회귀**: 버그를 고쳤다면 그 버그를 재현하던 케이스를 반드시 박제한다.

## 좋은 테스트의 조건
- **결정적**: 시간/난수/네트워크/순서에 의존하지 않는다. 필요하면 주입·목킹.
- **격리**: 테스트끼리 상태를 공유하지 않는다. 각자 셋업/정리.
- **하나의 이유로 실패**: 한 테스트는 한 가지를 검증한다.
- **읽히는 이름**: `test_<대상>_<상황>_<기대>` 형태로 의도를 드러낸다.
- **AAA 구조**: Arrange → Act → Assert.

## 절차
1. 대상 코드의 공개 동작과 계약을 파악한다.
2. 위 분류대로 케이스 목록을 만든다.
3. 테스트를 추가하고 실제로 실행해 통과를 확인한다(`run_tests`).
4. 새 테스트가 실제로 버그를 잡는지 확인하려면, 구현을 잠깐 깨뜨려 빨갛게
   되는지 본 뒤 되돌린다.
- 커버리지 수치를 위한 무의미한 테스트는 쓰지 않는다.
